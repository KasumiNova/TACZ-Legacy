# TACZ → TACZ-Legacy 完整迁移计划（1.12.2）

> 本计划按“先兼容、后重构、再优化”的策略推进，确保功能与生态稳定落地。

## A. 目标与验收硬标准

### A1. 美术素材可复用
- **目标**：优先复用 TACZ 现有贴图/模型/音效资源。
- **验收**：
  - 核心资源路径映射清单覆盖率 ≥ 95%
  - 无需二次导出的素材可直接加载
  - 仅对少量格式差异资源进行自动转换

### A2. 枪包/数据包完美兼容
- **目标**：既有枪包在 TACZ-Legacy 中无语义损失。
- **验收**：
  - 枪包字段兼容测试全通过
  - 默认行为与 TACZ 对齐（含边界值）
  - 发现差异时有兼容层而非直接 break

### A3. 整体功能一致
- **目标**：玩家侧功能行为、数值效果、交互反馈保持一致。
- **验收**：
  - 功能矩阵逐项对齐（开火、换弹、配件、命中、动画、音画反馈）
  - 联机场景核心链路通过（客户端/服务端一致）

### A4. 可扩展渲染管线
- **目标**：重构为阶段化、模块化渲染框架。
- **验收**：
  - 新增特效无需侵入核心渲染循环
  - 透明/后处理顺序可配置且稳定
  - 渲染调试开关与诊断指标可用

### A5. 程序架构与 MC 架构分离（可测试优先）
- **目标**：核心程序逻辑可脱离 MC 运行时进行大量单元测试，同时避免与原版完全切割导致维护成本激增。
- **验收**：
   - 领域层（状态机/计算逻辑）不直接依赖 `net.minecraft.*`
   - 关键逻辑具备稳定单测，核心模块覆盖率目标 ≥ 70%
   - 存在明确 MC 适配层，且适配层有契约测试验证

### Phase 4 — 渲染管线重构（重点）

### 4.1 架构设计
引入四层模型（参考 Kirino-Engine 理念并按 1.12 约束裁剪）：

1. **RenderData Layer**
   - 从逻辑线程生成不可变 `RenderSnapshot`
   - 不在渲染线程做重型计算

2. **RenderPass Layer**
   - 固定阶段：`PRE_UPDATE` → `UPDATE` → `OPAQUE` → `TRANSLUCENT` → `POST` → `OVERLAY`
   - 每个阶段可注册多个 `RenderFeature`

3. **RenderDispatcher Layer**
   - 统一排序、批处理、状态切换与 flush
   - 维护稳定的透明顺序与状态恢复

4. **RenderFeature Plugin Layer**
   - 枪口焰、镜面、弹壳、命中提示等作为插件注册
   - 插件通过 capability/接口声明依赖

### 4.1.1 Kirino 风格的最小可行复刻
1. `FramePhase` + `FramePhaseFSM`：强约束阶段顺序
2. `RenderPass` + `Subpass`：组合式渲染执行单元
3. `DrawCommand`：先描述“画什么”，再编译“怎么画”
4. `RenderExtensions`：通过注册表扩展 shader、post、debug HUD
5. `Headless/Graphics` 双世界视图：逻辑与图形职责显式分离

### 4.2 执行策略
- 引入 `RenderContext`（帧上下文）
- 逻辑与渲染解耦：仅传输快照数据
- 所有特性可按配置启停
- 提供 Debug HUD：pass 耗时、批次数、实体计数
- 提供“Vanilla 回退路径”：新管线关闭时可退回原有渲染路径

### 4.3 验收
- 新增一个视觉特性时，不修改核心 dispatcher 主循环
- 复杂场景下渲染顺序稳定，无明显闪烁/错层
- 诊断面板可定位性能热点
- 新旧双路径在功能开关下可切换且结果可比对

**里程碑 M4**：新渲染框架替换旧路径并支持插件扩展。

## C. 风险与应对

1. **1.20 → 1.12 API 差异过大**
   - 应对：先抽象领域层，平台差异放适配层
2. **枪包历史字段不一致**
   - 应对：兼容读取器 + 字段别名 + 默认值策略
3. **渲染重构期间回归风险高**
   - 应对：双路径开关（旧/新）+ 分阶段灰度切换
4. **联机不同步导致体验劣化**
   - 应对：建立关键链路时序测试与回放日志
